Network Change Assurance

Upgrade Assurance Programme

Moving NX-OS fabric upgrades from after-the-fact firefighting to a controlled, evidenced process — with the same control language we already use for audit.
Today we find out what an upgrade broke when users complain.
The goal is to know before the change window what is at risk, and within seconds after exactly what moved and how to put it back.

Four controls, positioned by when they act relative to the reload. Two run before the change window, two after — and a feedback loop turns every upgrade into input for the next one. Select a stage on the timeline.

Nothing here is a criticism of the team — it is what any team does without tooling. The point is that the knowledge lives in people's heads and eyes, at 3am, under time pressure.

How an upgrade goes today

Manual, reactive, unevidenced

  • Show-command output saved to Notepad, compared by eye across ~19 commands
  • One missing line in a 3,000-line config is invisible to a tired reader
  • Operational state (adjacencies, VTEP peers, prefix counts) rarely compared at all
  • Problems surface when an application team calls, not at the change window
  • Root cause takes hours of hunting; the fix depends on who is on shift
  • No durable record that the device was healthy before and after
Where we are heading

Controlled, evidenced, repeatable

  • Known risks for the target release identified before the change window
  • Every command compared deterministically — config and operational state
  • Scattered symptoms correlated into one probable root cause
  • Config loss comes back with paste-ready remediation
  • Engineer sign-off recorded per finding — an audit trail, not a memory
  • Each upgrade improves the rule base for the next one

On numbers: no percentage improvement is claimed on this page. Any figure quoted before we have run the process on our own fleet would be invented, and this programme's whole premise is that claims must be evidenced. The measures above are what we will baseline first.

Three steps. Step 1 exists and can be demonstrated today. Each step is useful on its own — nothing depends on finishing the whole programme.

Deliberate choice — we are not rebuilding Genie. Cisco has maintained pyATS/Genie for years and gives it away free. Rebuilding a parser library would cost us years and leave us permanently behind. We use it as the engine and spend our effort where nothing exists yet: turning a difference into a diagnosis and a fix.

No software cost and no licence to buy. Cisco pyATS/Genie is free and installed from the public Python package index. The ask is infrastructure and permissions only.

01 Compute

  • One small Linux VM — Ubuntu LTS is the simplest choice
  • Cisco's stated minimum is 1 vCPU and 1 GB RAM; we would ask for 2 vCPU / 4 GB RAM / ~40 GB disk to run it comfortably as a shared service
  • Windows is not supported by pyATS — Cisco's own guidance is Linux, macOS, Docker, or WSL

02 Software

  • Python 3 (Cisco's getting-started guide lists 3.7–3.10; current packages ship for newer versions too, so we confirm the exact version against the release notes at install time)
  • pyATS + Genie, installed with pip
  • Ubuntu build packages: python3-venv, python3-dev, build-essential
  • A lightweight Python web service to serve the existing dashboard

03 Network access

  • Most likely blocker. The host needs outbound HTTPS to the Python package index and GitHub to install and update
  • If external package access is blocked, an internal package mirror works equally well
  • Inbound: engineers' browsers reach the service on an internal port

04 Access and permissions

  • Administrator rights on the VM to install packages
  • If we pilot on a laptop instead: approval to enable WSL
  • Later, for automated bug verification: a Cisco account with Bug Search access

05 What we do not need

  • No licence, no procurement, no vendor contract
  • No SSH access from the service to production switches — captures are pasted or uploaded as text, so the tool never touches a live device
  • No Python skills for the team using it — they use a browser
  • No data leaves our network: Cisco states the pyATS ecosystem never collects or streams data back to Cisco

06 Data handling

  • Captures include full running-config, which contains sensitive material (community strings, hashes)
  • Store snapshots on internal storage with defined retention; never in a public repository
  • Step 1 avoids this entirely — nothing is stored or transmitted at all
RECOMMENDED FIRST

Option A — pilot on a laptop with WSL

Costs nothing and needs no server. Only ask: permission to enable WSL. We prove the full Genie chain works on our own captures, then use that result to justify the VM.

THEN

Option B — shared internal VM

Once proven, move to a small Linux VM so the whole team uses one service with a consistent rule base and a shared record of every upgrade.

A tool that explains why something broke is only useful if we can tell the difference between a fact and a guess. Every finding carries its source and one of two stamps.

VERIFIED BY DESIGN

Deterministic. Either a plain difference between our own two captures, or a consequence of how the protocol is defined. Safe to act on.

e.g. a config line present before and missing after
e.g. VXLAN encapsulation overhead against the interface MTU

VERIFY REQUIRED

Anything that depends on a specific release or platform behaving a certain way. The tool flags it and points the engineer at the vendor's own record — it never states it as fact.

e.g. "this release resets that setting during migration"
→ confirm on the Cisco Bug Search Tool and the target release notes

Why this distinction exists — a real example from building it

During development the tool produced a confident explanation that a specific NX-OS release reset interface MTU during config migration. It read plausibly and it fitted the symptoms perfectly.

We checked it against the actual Cisco release notes for that version — open issues, resolved issues and known issues. No such caveat exists. The explanation was invented. What the release notes did contain were several genuine post-upgrade drift bugs, so the underlying pattern was real; the specific attribution was not.

That is precisely the failure mode this programme has to guard against, because a plausible wrong cause sends an engineer down the wrong path during a change window. The two-stamp model is the direct result: deterministic findings are presented as facts, and version-specific causes are presented as claims to be verified against the vendor's own record before anyone acts on them.

Bottom line for governance: the tool never applies a change. It presents the difference, the reasoning, the source of that reasoning, and a proposed fix. A named engineer reviews each finding and records the decision — the same accept / flag / roll-back discipline we already apply to any other control.
